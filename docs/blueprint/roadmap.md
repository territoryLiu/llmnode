# 未来规划

## 0. 文档定位

这份文档只写未来，不重复当前状态。  
它的职责是把“下一步该做什么、为什么这样排、哪些现在不做”说清楚。

它不负责：

- 复述当前已落地内容，那是 [current.md](current.md)
- 记录阶段演进，那是 [history.md](history.md)
- 展开重大设计方案，那是相关 `docs/superpowers/specs/*.md`
- 记录具体实施拆分，那是相关 `docs/superpowers/plans/*.md`

## 1. 使用规则

这份路线图只保留“尚未完全落地”的事项。  
一项工作一旦从未来规划变成当前事实，就应该：

1. 从本文件移出
2. 回流到 [current.md](current.md)
3. 如有阶段意义，再补到 [history.md](history.md)
4. 同步相关 `contracts / process / glossary`

## 2. 当前规划假设

当前路线图建立在下面几个假设上：

- 单后端正式路径已经成立，当前重点不是从零起网关
- 控制入口已经收口，当前重点不是再补 shell 包装层
- 当前瓶颈集中在多后端扩展、契约收敛、管理台补厚和文档治理
- 新能力不能跳过契约和控制面治理直接上功能壳子，否则后面返工成本更高
- 当前阶段仍以单机为主，不优先进入多节点和 K8s
- V3 的有效未来内容应收敛为“三后端统一 Docker 化”
- V4 的有效未来内容应收敛为“节点平台化控制面”
- 更细的设计展开和实施计划应统一进入 `docs/superpowers/*`
- 三后端 Docker 化的含义是：`vLLM / llama.cpp / SGLang` 各自使用官方 Docker 镜像或官方容器方案，Python 控制面只负责 Docker 编排、状态读取与契约治理

## 2.1 来自 V3 的未来设计收敛

V3 当前仍有效的未来方向已经收敛为：

- 三后端统一抽象：
  - `vLLM`
  - `llama.cpp`
  - `SGLang`
- 三后端统一 Docker 化
- 每个后端各自一个官方 Docker
- `gateway-api / node-agent / web-console / control-plane` 围绕三后端继续对齐
- Python 控制面只与 Docker 交互，不再承担宿主机原生部署、手工编译或多套本地脚本治理

这些方向不再需要和 `roadmap.md` 并列维护第二份未来总纲，而应继续以内聚方式留在本文件。

## 2.2 来自 V4 的未来设计收敛

V4 当前仍有效的未来方向已经收敛为：

- 从“单机控制面”继续走向“节点平台化控制面”
- 引入更明确的节点对象与运行对象：
  - `Node`
  - `ModelArtifact`
  - `RuntimeProfile`
  - `LogicalModelRoute`
  - `RuntimeInstance`
- 继续保持：
  - `gateway-api` 负责推理热路径
  - 控制面负责节点事实源
  - `node-agent` 负责执行源
- 为未来 1~3 节点的小规模扩展留边界

这些内容当前更适合作为路线图中的后续阶段，而不应单独与 `roadmap.md` 竞争“未来规划入口”。

## 3. P0：先稳定当前正式路径

### 3.1 稳定 `vLLM` 正式主路径

重点：

- 保持 `vLLM` 路径继续可用
- 稳定 `python -m llmnode.control` 的输出和行为
- 提高 `doctor / logs / status` 对真实问题的定位能力

为什么优先：

- 当前所有多后端扩展都建立在正式单后端路径稳定之上
- 如果主路径还不稳，三后端扩展只会增加维护成本

### 3.2 补厚管理台

重点：

- 管理台与控制面输出保持一致
- 展示更多后端状态、日志入口、事件摘要
- 提升配置与诊断体验

为什么优先：

- 当前管理台已经存在，但仍更偏观察层
- 后续如果要接三后端，它必须先成为可靠的治理入口

## 4. P1：完成三后端抽象

### 4.1 抽象统一 `BackendDriver`

重点：

- 保留现有 `vLLM` 驱动行为
- 抽出统一容器规格、健康检查、状态采样接口
- 为 `llama.cpp` 和 `SGLang` 预留一致控制面
- 所有驱动默认以 Docker 容器为运行单元，不再把宿主机原生启动当成正式主路径

### 4.2 扩展配置与路由模型

重点：

- `backend_type` 支持：
  - `vllm`
  - `llama_cpp`
  - `sglang`
- 扩展 profile、容器镜像、端口和健康检查字段
- 让管理接口、数据库和前端都能理解三种后端
- 明确“镜像、命令、卷挂载、端口、健康检查、模型目录映射”是三后端统一契约的一部分

## 5. P2：落地 `llama.cpp`

重点：

- 通过 Docker 启动 `llama.cpp`
- 明确其 profile、容器参数和健康检查规则

为什么单列：

- `llama.cpp` 与 `vLLM` 的资源模型和部署侧重点差异较大
- 它是三后端里最需要单独处理资源画像和容器参数的一类

## 6. P3：落地 `SGLang`

重点：

- 通过 Docker 启动 `SGLang`
- 明确与 `vLLM` 的角色边界
- 在控制面、日志、健康检查和管理台中成为正式一等后端

为什么单列：

- `SGLang` 不是简单替代 `vLLM`
- 它更像第二类 GPU 服务化推理后端，需要单独收敛定位和运行策略

## 7. P4：文档系统第二轮收口

重点：

- 维持“不再保留 `docs/blueprintV3.md` / `docs/blueprintV4.md`”这一入口约束
- 决定 `docs/knowledge/*` 的长期定位
- 规范 `docs/superpowers/*` 的保留与删除策略

为什么重要：

- 当前文档系统虽然已分层，但厚度和职责边界还可继续提升
- 如果不继续收口，后续信息会重新散回旧文件

## 8. P5：节点平台化预留

重点：

- 在不引入完整商业平台复杂度的前提下，把当前控制面逐步抽象成更清晰的节点平台骨架
- 逐步明确：
  - 节点对象
  - 运行实例对象
  - artifact / profile / route 之间的关系
- 让后续如果需要接入上层平台时，不必重做本地节点层

为什么现在只是预留：

- 当前主问题仍是单机三后端落地与治理
- 如果过早进入节点平台化，很容易分散当前正式路径收口的注意力

## 9. 当前不优先做的事

- 多节点平台
- 分布式调度
- K8s 编排
- 模型自动下载
- 自动格式转换流水线
- 大而全的推荐系统式前端壳

## 10. 旧蓝图与路线图的关系

- V1 / V2 已经是历史阶段，应进入归档层
- V3 / V4 不再保留为独立蓝图文件
- V3 / V4 的未来规划应回流进本文件
- V3 / V4 的设计展开与实施计划应进入 `docs/superpowers/*`

## 11. 下一步怎么使用这份文档

- 如果一项工作还没落地，但需要判断优先级，先看这里
- 如果一项工作已经落地，就不应继续停留在这里
- 如果一项工作需要设计展开，转去相关 `specs`
- 如果一项工作需要执行拆分，转去相关 `plans`
