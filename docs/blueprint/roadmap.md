# 未来规划

## 0. 文档定位

这份文档只写未来，不重复当前状态。  
它的职责是把“下一步该做什么、为什么这样排、哪些现在不做”说清楚。

它不负责：

- 复述当前已落地内容，那是 [current.md](current.md)
- 记录阶段演进，那是 [history.md](history.md)
- 展开重大设计方案，那是相关 `docs/superpowers/specs/*.md`
- 记录具体实施拆分，那是相关 `docs/superpowers/plans/*.md`

## 1. 使用规则

这份路线图只保留“尚未完全落地”的事项。  
一项工作一旦从未来规划变成当前事实，就应该：

1. 从本文件移出
2. 回流到 [current.md](current.md)
3. 如有阶段意义，再补到 [history.md](history.md)
4. 同步相关 `contracts / process / glossary`

## 2. 当前未来边界

当前路线图只保留下面几类尚未完全落地的事项：

- 启动窗口期与就绪性加固
- 平台级用量与 Token 指标采集
- API Key 管理与管理台补厚
- 节点平台化预留
- 仍未进入正式真相层的未来设计展开

默认前提：

- 当前正式默认后端仍是 `vLLM`
- 当前阶段仍以单机为主，不优先进入多节点和 K8s
- 三后端 Docker 化已经成立，未来重点不再是回到旧控制入口或旧蓝图文件
- 更细的设计展开和实施计划统一进入 `docs/superpowers/*`

## 3. P0：启动窗口期与就绪性加固

重点：

- 把“后端仍在加载 / 热身中”明确作为标准可重试状态透传给客户端，而不是继续暴露误导性的 `500`
- 对 Claude Code 等流式客户端优先返回标准信号：
  - `503 Service Unavailable`
  - `Retry-After: 5` 或 `10`
  - JSON detail 明确标注 `backend warming up` / `backend not stream-ready`
- 把 readiness 从单一 `ready` 拆成至少两层：
  - `http_ready`：后端 HTTP 已可访问
  - `inference_ready`：首个实际推理与流式输出已稳定
- 后端启动后增加一次极小探针请求，再把 Agent 状态提升为 ready，而不是只看 `/v1/models`
- 单独记录并可检索这些事件：
  - `warming_up`
  - `stream_not_ready`
  - `backend_recovered`
- 回流文档真相，避免默认 profile 与排障文档继续引用过期的 `35B-A3B-FP8` 默认结论
- 顺手补齐当前偏松的默认安全边界：
  - 网关默认 `dev-key` 不应继续作为正式默认
  - 后端监听范围应从“默认对外可达”收紧到更安全的默认值

为什么重要：

- 当前已确认存在启动窗口期：`/v1/models` 已返回 `200`，首个 Claude 流式请求仍可能断开
- 当前 ready 判断仍然过粗，只看了 `/v1/models`，会让客户端把“可探活”误判成“可稳定推理”
- 先把错误语义、恢复语义和日志语义理顺，后续多后端治理、诊断与自动恢复才有稳定基础

## 4. P1：平台级用量与 Token 指标采集

重点：

- 在现有基础 `request_metrics` 之上，补齐平台级请求用量账本，而不是只停留在请求数、成功率和基础吞吐
- 重点记录并可聚合：
  - 输入 tokens
  - 输出 tokens
  - 缓存命中 tokens
  - 缓存未命中 tokens
  - 总 tokens
  - 各请求的延迟、吞吐和协议类型
- 聚合维度至少覆盖：
  - 模型
  - 后端类型
  - API Key
  - 时间粒度（日 / 月 / 年）
  - 必要时再补 endpoint / 协议维度
- 管理台需要提供“平台视角”的统计页，而不只是请求明细表，可参考 `skill-usage` 的事件聚合与趋势展示方式，但数据模型要面向模型网关场景
- 指标设计要兼容不同后端 usage 字段差异：
  - 原样接收后端已返回的 usage / cache 统计
  - 后端未提供时明确记为 unknown / unavailable，而不是伪造为 0
  - 为后续成本估算、配额、限流与审计保留扩展字段

为什么重要：

- 一个可用的大模型管理平台，不能只有“请求是否成功”，还要能回答“谁在用、用了多少、缓存是否生效、哪条后端最贵或最忙”
- 当前基础指标已经落地，但还不足以支撑按模型 / 后端 / 密钥 / 时间维度做运营、容量和审计判断
- 这部分做好后，后续无论接入更多模型还是上层平台，都不用重做底层账本

## 5. P1：API Key 管理与管理台补厚

已完成（2026-05-14）：

- API Key 管理页已补齐：名称、脱敏 `masked_key`、显示/隐藏密钥、一键复制、状态切换、创建/禁用/删除
- `/admin/keys` 列表接口返回 `masked_key` 和 `usage_summary`（`total_requests` / `total_tokens`）
- `/admin/overview/readiness` 端点统一下发 Base URL（本机 `127.0.0.1:4000` / 局域网 `10.18.90.100:4000`）
- 管理台 Key 视图顶部展示可复制的 Base URL 卡片
- 新建密钥当次显示完整 secret，支持显示/隐藏切换和复制

剩余工作：

- Key 维度的统计应与上一节的平台用量账本打通，避免”有 key 列表，但看不到该 key 用量”这种割裂体验
- 密钥列表可搜索、可筛选的增强体验
- 这是后续做配额、团队协作、审计与多客户端接入的基础面

为什么重要：

- 当前底层已有 key 存储与基础管理接口，但还不等于”用户可长期依赖的正式管理能力”
- 如果 key 无法稳定持久化、无法按 key 看用量、无法明确给出 Base URL，管理台就还不具备交付价值

## 6. P2：节点平台化预留

重点：

- 在不引入完整商业平台复杂度的前提下，把当前控制面逐步抽象成更清晰的节点平台骨架
- 逐步明确：
  - 节点对象
  - 运行实例对象
  - artifact / profile / route 之间的关系
- 让后续如果需要接入上层平台时，不必重做本地节点层

为什么现在只是预留：

- 当前主问题仍是单机三后端落地与治理
- 如果过早进入节点平台化，很容易分散当前正式路径收口的注意力

## 7. 当前不优先做的事

- 多节点平台
- 分布式调度
- K8s 编排
- 模型自动下载
- 自动格式转换流水线
- 大而全的推荐系统式前端壳
