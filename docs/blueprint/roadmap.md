# 未来规划

## 0. 文档定位

这份文档只保留尚未完全落地的事项。  
它的职责是把“接下来最该补什么、为什么这样排、哪些明确不做”说清楚。

它不负责：

- 复述当前已落地内容，那是 [current.md](current.md)
- 记录阶段演进，那是 [history.md](history.md)
- 展开重大设计方案，那是相关 `docs/superpowers/specs/*.md`
- 记录具体实施拆分，那是相关 `docs/superpowers/plans/*.md`

## 1. 使用规则

一项工作一旦从未来规划变成当前事实，就应该：

1. 从本文件移出
2. 回流到 [current.md](current.md)
3. 如有阶段意义，再补到 [history.md](history.md)
4. 同步相关 `contracts / process / glossary`

## 2. 当前未来边界

当前路线图只保留下列尚未完全落地的事项：

- 配置真相源与文档口径收口
- route 产品边界澄清与管理闭环
- 节点平台化预留
- 已落地管理面的继续补厚项

默认前提：

- 当前阶段仍以单机为主，不优先进入多节点和 K8s
- 三后端 Docker 化已经成立，未来重点不是回到旧控制入口
- 当前使用什么后端、模型与参数，由当前激活的 profile 和对应配置决定
- 更细的设计展开和实施计划统一进入 `docs/superpowers/*`

## 3. 下一阶段连续任务

下面 7 项按推荐顺序执行，前一项不收口，后一项通常不要提前展开。

### 任务 1：配置真相源收口

目标：

- 彻底去掉“默认模型”叙事
- 统一文档、代码注释、测试基线到“当前激活 profile / 当前配置决定运行真相”

完成条件：

- `README.md`
- `docs/blueprint/current.md`
- `docs/blueprint/history.md`
- `docs/contracts/backend-routing.md`
- 相关测试描述

这些位置不再把某个模型写成脱离配置存在的“正式默认模型”。

为什么优先：

- 这是当前最直接的真相源分叉点
- 不先收口，后续 route 与 profile 讨论会持续混乱

### 任务 2：测试基线与配置加载语义对齐

目标：

- 清掉依赖旧默认值假设的测试
- 把测试改成验证“配置是否正确生效”，而不是验证“某个模型是不是默认值”

完成条件：

- `tests/test_smoke.py` 等配置相关测试不再断言固定默认模型名
- 配置加载测试围绕 `active_backend_profile`、profile 文件解析和运行参数映射展开

为什么排第二：

- 这是把真相源收口落实到工程基线的最小闭环
- 不做这一步，后面无法稳定判断文档和代码是否一致

### 任务 3：明确 route 当前正式边界

目标：

- 把 route 的当前正式定位写清楚
- 明确当前是“配置派生 + 运行态编辑”，还不是完整模型注册中心

完成条件：

- `current.md`
- `backend-routing.md`
- 必要时 `README.md`

明确写出：

- route 初值来自当前激活 profile
- 启动时会 seed 到 `model_routes`
- 当前管理台可以编辑已有 route
- 当前还没有完整新增 / 删除 / 长期持久化闭环

为什么排第三：

- 这是判断下一阶段要不要进入 route 平台化的前提

### 任务 4：决定 route 是继续轻量运行态，还是升级成长期持久化管理

目标：

- 做产品边界决策，而不是继续混用两套说法

两种方向只能选一条主路：

- 轻量路：维持“配置派生 + 运行态编辑”
- 平台路：升级为完整 route 注册中心

完成条件：

- 在正式文档中明确选型
- 若选平台路，补 `spec`
- 若选轻量路，补使用边界与限制说明

为什么排第四：

- 这一步决定后续代码投入方向
- 不拍板，新增 / 删除 / 持久化能力会一直悬空

### 任务 5：如果进入平台路，补 `admin/models` 的新增能力

目标：

- 允许创建新的 logical route，而不只是编辑已有 route

完成条件：

- 后端有新增 route 接口
- 管理台有创建入口
- 新增 route 有最小校验边界

为什么排第五：

- 这是从“编辑现有项”走向“可管理对象”的第一步

### 任务 6：如果进入平台路，补 route 删除与持久化策略

目标：

- 明确 route 删除语义
- 明确启动 seed 与手工维护 route 的冲突解决方式

完成条件：

- 删除接口与前端入口
- 文档写清楚 catalog、SQLite、管理台修改三者关系
- 启动同步策略可预测，不再隐含清表副作用

为什么排第六：

- 没有删除和持久化策略，route 新增能力会变成半成品

### 任务 7：节点平台化预留

目标：

- 在保持单机前提下，继续收口对象边界
- 逐步明确 `node / instance / artifact / profile / route` 的关系

完成条件：

- 至少完成文档级对象边界定义
- 必要时再进入代码层抽象

为什么排最后：

- 这是结构预留，不是当前主链路阻塞项
- 只有在配置真相和 route 边界稳定后，这项工作才有意义

## 4. 当前不优先做的事

- 多节点平台
- 分布式调度
- K8s 编排
- 模型自动下载
- 自动格式转换流水线
- 成本估算
- 配额 / 限流联动
- 大而全的推荐系统式前端壳

## 5. 旧蓝图与路线图的关系

- V1 / V2 已经是历史阶段，应进入归档层
- V3 / V4 不再保留为独立蓝图文件
- 未来规划统一看本文件
- 设计展开与实施计划统一进入 `docs/superpowers/*`

## 6. 下一步怎么使用这份文档

- 如果一项工作已经落地，就不应继续停留在这里
- 如果一项工作只是状态澄清，先回流 `current / contracts / process`
- 如果一项工作要进入中大规模实现，再补对应 `spec / plan`
- 如果要安排开发顺序，默认按“任务 1 → 任务 7”的顺序推进
