# 未来规划

## 0. 文档定位

这份文档只保留尚未完全落地的事项。  
它的职责是把“接下来最该补什么、为什么这样排、哪些明确不做”说清楚。

它不负责：

- 复述当前已落地内容，那是 [current.md](current.md)
- 记录阶段演进，那是 [history.md](history.md)
- 展开重大设计方案，那是相关 `docs/superpowers/specs/*.md`
- 记录具体实施拆分，那是相关 `docs/superpowers/plans/*.md`

## 1. 使用规则

一项工作一旦从未来规划变成当前事实，就应该：

1. 从本文件移出
2. 回流到 [current.md](current.md)
3. 如有阶段意义，再补到 [history.md](history.md)
4. 同步相关 `contracts / process / glossary`

## 2. 当前未来边界

当前路线图只保留下列尚未完全落地的事项：

- route 管理闭环补厚
- 节点平台化预留
- 已落地管理面的继续补厚项

默认前提：

- 当前阶段仍以单机为主，不优先进入多节点和 K8s
- 三后端 Docker 化已经成立，未来重点不是回到旧控制入口
- 当前使用什么后端、模型与参数，由当前激活的 profile 和对应配置决定
- 更细的设计展开和实施计划统一进入 `docs/superpowers/*`

## 3. 下一阶段连续任务

下面 2 项按推荐顺序执行，前一项不收口，后一项通常不要提前展开。

### 任务 1：补齐 route phase 2 管理闭环

目标：

- 在 phase 1 已落地的长期 route 注册表基础上，继续补齐治理体验与可观测性
- 让 `stale`、来源边界和管理动作对操作者更可预测

完成条件：

- 在已有治理提示、overview 摘要、startup seed 事件基础上，继续补齐 route 管理动作语义
- 明确 `stale + profile_seed` route 后续允许和不允许的治理动作
- 必要时再决定是否新增更多事件、摘要或诊断聚合，而不是平铺新接口
- `current / contracts / process` 对 phase 2 行为继续回流

为什么排第一：

- route 的基础可观测性与来源边界已经成立，接下来最值得补的是动作闭环，而不是重复建设展示层

### 任务 2：节点平台化预留

目标：

- 在保持单机前提下，继续收口对象边界
- 逐步明确 `node / instance / artifact / profile / route` 的关系

完成条件：

- 至少完成文档级对象边界定义
- 必要时再进入代码层抽象

为什么排第二：

- 这是结构预留，不是当前主链路阻塞项
- 只有在 route phase 1/2 边界稳定后，这项工作才有意义

## 4. 当前不优先做的事

- 多节点平台
- 分布式调度
- K8s 编排
- 模型自动下载
- 自动格式转换流水线
- 成本估算
- 配额 / 限流联动
- 大而全的推荐系统式前端壳

## 5. 旧蓝图与路线图的关系

- V1 / V2 已经是历史阶段，应进入归档层
- V3 / V4 不再保留为独立蓝图文件
- 未来规划统一看本文件
- 设计展开与实施计划统一进入 `docs/superpowers/*`

## 6. 下一步怎么使用这份文档

- 如果一项工作已经落地，就不应继续停留在这里
- 如果一项工作只是状态澄清，先回流 `current / contracts / process`
- 如果一项工作要进入中大规模实现，再补对应 `spec / plan`
- 如果要安排开发顺序，默认按“任务 1 → 任务 2”的顺序推进
