# 未来规划

## 0. 文档定位

这份文档只写未来，不重复当前状态。  
它的职责是把“下一步该做什么、为什么这样排、哪些现在不做”说清楚。

它不负责：

- 复述当前已落地内容，那是 [current.md](current.md)
- 记录阶段演进，那是 [history.md](history.md)
- 展开重大设计方案，那是相关 `docs/superpowers/specs/*.md`
- 记录具体实施拆分，那是相关 `docs/superpowers/plans/*.md`

## 1. 使用规则

这份路线图只保留“尚未完全落地”的事项。  
一项工作一旦从未来规划变成当前事实，就应该：

1. 从本文件移出
2. 回流到 [current.md](current.md)
3. 如有阶段意义，再补到 [history.md](history.md)
4. 同步相关 `contracts / process / glossary`

## 2. 当前规划假设

当前路线图建立在下面几个假设上：

- 单后端正式路径已经成立，当前重点不是从零起网关
- 控制入口已经收口，当前重点不是再补 shell 包装层
- 控制面诊断能力已增强（2026-05-12），当前瓶颈转移到文档治理和性能指标采集
- 新能力不能跳过契约和控制面治理直接上功能壳子，否则后面返工成本更高
- 当前阶段仍以单机为主，不优先进入多节点和 K8s
- V3 的有效未来内容应收敛为”三后端统一 Docker 化”
- V4 的有效未来内容应收敛为”节点平台化控制面”
- 更细的设计展开和实施计划应统一进入 `docs/superpowers/*`
- 三后端 Docker 化的含义是：`vLLM / llama.cpp / SGLang` 各自使用官方 Docker 镜像或官方容器方案，Python 控制面只负责 Docker 编排、状态读取与契约治理

## 2.1 来自 V3 的未来设计收敛

V3 有效方向已经落地：

- 三后端统一抽象（`BackendDriver` Protocol + 三套实现）已完成
- 三后端统一 Docker 化（ContainerSpec / 官方镜像 / Python 控制面编排）已完成
- `gateway-api / node-agent / web-console / control-plane` 围绕三后端的代码对齐已完成
- **三后端线上联调验证已完成（2026-05-12）**
- **控制面诊断能力增强已完成（2026-05-12）**：
  - `doctor` 命令支持三后端特定检查、GPU 信息、模型格式检测、智能建议
  - `status` 命令支持容器详细信息、推理参数展示、6 种栈状态
  - `logs` 命令支持实时跟踪、错误高亮、关键词搜索
  - Agent 服务暴露诊断 API 端点（`/admin/diagnostics/*`）
  - 管理台前端已对齐三后端状态展示

## 2.2 来自 V4 的未来设计收敛

V4 当前仍有效的未来方向已经收敛为：

- 从“单机控制面”继续走向“节点平台化控制面”
- 引入更明确的节点对象与运行对象：
  - `Node`
  - `ModelArtifact`
  - `RuntimeProfile`
  - `LogicalModelRoute`
  - `RuntimeInstance`
- 继续保持：
  - `gateway-api` 负责推理热路径
  - 控制面负责节点事实源
  - `node-agent` 负责执行源
- 为未来 1~3 节点的小规模扩展留边界

这些内容当前更适合作为路线图中的后续阶段，而不应单独与 `roadmap.md` 竞争“未来规划入口”。

## 3. P0：文档系统第二轮收口

重点：

- 维持”不再保留 `docs/blueprintV3.md` / `docs/blueprintV4.md`”这一入口约束
- 决定 `docs/knowledge/*` 的长期定位
- 规范 `docs/superpowers/*` 的保留与删除策略

为什么优先：

- 控制面诊断能力已增强完成，当前瓶颈转移到文档治理
- 当前文档系统虽然已分层，但厚度和职责边界还可继续提升
- 如果不继续收口，后续信息会重新散回旧文件

## 4. P1：性能指标采集

重点：

- 在不引入重型监控系统的前提下，采集基础性能指标
- 请求总数、成功率、平均延迟、P95/P99 延迟
- 吞吐量（tokens/s）、队列长度
- 通过 Agent 服务暴露 `/admin/diagnostics/metrics` 端点

为什么重要：

- 当前只有健康检查，缺少性能观测维度
- 性能指标对于调优和容量规划很重要

## 5. P2：节点平台化预留

重点：

- 在不引入完整商业平台复杂度的前提下，把当前控制面逐步抽象成更清晰的节点平台骨架
- 逐步明确：
  - 节点对象
  - 运行实例对象
  - artifact / profile / route 之间的关系
- 让后续如果需要接入上层平台时，不必重做本地节点层

为什么现在只是预留：

- 当前主问题仍是单机三后端落地与治理
- 如果过早进入节点平台化，很容易分散当前正式路径收口的注意力

## 6. 当前不优先做的事

- 多节点平台
- 分布式调度
- K8s 编排
- 模型自动下载
- 自动格式转换流水线
- 大而全的推荐系统式前端壳

## 7. 旧蓝图与路线图的关系

- V1 / V2 已经是历史阶段，应进入归档层
- V3 / V4 不再保留为独立蓝图文件
- V3 / V4 的未来规划应回流进本文件
- V3 / V4 的设计展开与实施计划应进入 `docs/superpowers/*`

## 8. 下一步怎么使用这份文档

- 如果一项工作还没落地，但需要判断优先级，先看这里
- 如果一项工作已经落地，就不应继续停留在这里
- 如果一项工作需要设计展开，转去相关 `specs`
- 如果一项工作需要执行拆分，转去相关 `plans`
