# Web Console 全中文与中英切换改造设计

## 1. 目标

当前 `web-console` 存在三个问题：

- 导航、页面标题、按钮、表格和空态文案中英混杂
- 没有统一语言切换机制，后续继续补文案会越来越散
- 左上角品牌区仍是 `Nexus Console`，与当前 `llmnode / LlmNode` 品牌不一致

本设计的目标是：

- 默认把 `web-console` 所有可见 UI 文案切到中文
- 提供一个可点击的中英切换按钮，使整个控制台切到英文
- 将左上角 logo 更换为用户提供的品牌图，旁边文字改为 `LlmNode`

本设计不解决：

- 多语言持久化到服务端账号偏好
- 三种及以上语言扩展
- 完整第三方 i18n 框架接入
- 控制台整体视觉风格重做

## 2. 适用范围

本次“统一”按整站范围处理，而不是只改单页。

需要进入双语词典和切换控制的内容包括：

- 侧边导航文案
- 顶部页标题、副标题、连接状态
- 所有页面标题、说明文案、按钮文案
- 统计卡片标题
- 表格表头
- 表单标签与 placeholder
- 空态、加载态、成功提示、警告提示
- 前端自行生成的状态词

不强制翻译的内容包括：

- 后端原始返回的模型名、容器名、镜像名、请求 ID、IP、错误原文
- JSON 原始 inspect 数据
- 数据库中已有的自由文本 note

原则是：

- 前端可控文案统一双语
- 后端事实值尽量原样展示，避免误翻译或破坏排障语义

## 3. 设计原则

- 轻量优先：不引入额外 i18n 依赖
- 单一文案源：所有 UI 文案集中管理，避免页面内散落 `locale === 'zh' ? ... : ...`
- 默认中文：控制台首次进入默认展示中文
- 一键切换：切换按钮作用于整个控制台，而不是单页局部
- 最小侵入：尽量复用当前 `AppProvider` 和布局结构，不重构页面层级
- 品牌统一：品牌区、标题区和页面细节统一使用 `LlmNode`

## 4. 方案选择

本次采用“轻量本地词典 + 全局 locale 状态”的方案。

### 4.1 推荐方案

在 `web-console/src` 内新增：

- 一个 `locale` 全局状态，类型为 `zh | en`
- 一个集中式 `translations` 词典
- 一个前端取词函数，例如 `t(key)`

所有页面通过统一文案 key 读取文本。

优点：

- 不引入新依赖
- 改动集中，后续维护成本低
- 与当前项目体量匹配

### 4.2 不采用的方案

不采用“每个页面手写三元表达式”的方案，因为：

- 重复度高
- 文案难统一
- 后续继续扩页会迅速失控

不采用“引入完整 i18n 库”的方案，因为：

- 当前没有复杂插值、复数规则和命名空间需求
- 会让这次改造超过实际需要

## 5. 架构与落点

### 5.1 状态落点

语言状态放在现有前端全局状态层中，跟随 `AppProvider` 管理。

建议新增字段：

- `locale: 'zh' | 'en'`
- `setLocale(locale)`
- `toggleLocale()`
- `t(key)`

这样所有页面都能通过 `useAppContext()` 拿到当前语言和取词能力。

### 5.2 词典落点

新增一个集中词典文件，建议放在：

- `web-console/src/i18n.ts`

或等价的局部目录中。

词典按功能分组，至少覆盖：

- `layout`
- `overview`
- `usage`
- `keys`
- `models`
- `schedule`
- `status`
- `common`

key 命名要稳定、语义化，例如：

- `layout.nav.overview`
- `layout.connection.live`
- `overview.quickActions.restart`
- `keys.form.create`

### 5.3 组件使用方式

页面组件不直接拼硬编码中英文，而是统一通过 `t()` 读取。

允许保留少量运行态映射函数，例如：

- 将 `ok / error / rejected` 映射为当前语言显示值
- 将 `ready / recovering / failed` 映射为当前语言显示值

但这类映射也应尽量收进统一 helper，而不是散落在页面里。

## 6. 交互设计

### 6.1 默认语言

首次进入默认使用中文。

本次不要求语言选择写入浏览器存储或服务端配置。  
如果实现成本很低，可以写入 `localStorage` 作为增强，但不是本设计的硬要求。

### 6.2 切换按钮

在顶部操作区增加语言切换按钮。

交互要求：

- 当前为中文时，按钮应明确提示可切换到英文
- 当前为英文时，按钮应明确提示可切换到中文
- 切换后整站可见 UI 文案同步变化
- 不要求局部动画，但不能出现刷新整页或闪烁错位

按钮文案可以采用下面任一稳定形式：

- `中文 | EN`
- `中 / EN`
- `切换到 English` / `Switch to 中文`

实现时选一种清晰且与现有顶部区风格一致的形式即可。

### 6.3 页面标题

顶部当前使用 `Nexus <page>` 形式。

本次调整为：

- 品牌主名统一为 `LlmNode`
- 页面标题显示当前页面的本地化名称

例如中文下可展示：

- `LlmNode 总览`
- `LlmNode 调度`

英文下可展示：

- `LlmNode Overview`
- `LlmNode Schedule`

## 7. 品牌区改造

### 7.1 左上角品牌

将当前侧边栏左上角的：

- `Φ`
- `Nexus Console`

替换为：

- 用户提供的 logo 图片
- 文案 `LlmNode`

### 7.2 资源使用

仓库当前已存在：

- `web-console/logo.png`

实现时优先复用该资源路径接入到布局中。  
如果该文件内容与用户提供图一致，则不需要新增资源。  
如果内容不一致，后续实现阶段再决定是替换资源还是新增命名更清晰的品牌文件。

### 7.3 品牌显示要求

- logo 在浅色玻璃态背景下清晰可见
- 文字固定为 `LlmNode`
- 不再出现 `Nexus Console` 作为主品牌名
- 如果需要副标题，应使用与 `llmnode` 当前定位一致的表达，不能再沿用旧品牌口径

## 8. 页面改造范围

本次实现至少覆盖下面文件中的所有可见 UI 文案：

- `web-console/src/components/Layout.tsx`
- `web-console/src/pages/OverviewView.tsx`
- `web-console/src/pages/UsageRecordsView.tsx`
- `web-console/src/pages/ApiKeysView.tsx`
- `web-console/src/pages/ModelRoutesView.tsx`
- `web-console/src/pages/ScheduleView.tsx`
- `web-console/src/pages/SystemStatusView.tsx`

其中：

- `Layout.tsx` 负责导航、品牌、顶部标题、连接态、语言切换入口
- 各页面负责本页内容双语化

## 9. 特殊文案边界

### 9.1 状态枚举

下列后端或前端状态应做显示层映射：

- `ok / error / rejected`
- `ready / recovering / failed / unknown`
- `running / active / disabled / manual`
- `true / false` 的展示值

要求：

- 原始值如果仍参与逻辑判断，内部继续保留原值
- 只在展示层转换为当前语言文本

### 9.2 时间与数字

时间、数字、百分比、容量不要求为中英文分别设计不同格式。  
本次可以继续复用浏览器现有格式化逻辑；如果已有 `toLocaleString()`，可根据当前 `locale` 决定语言标签，但这不是必须项。

### 9.3 占位符与提示语

placeholder、按钮 loading 文案、复制成功提示等，也算“统一”范围，必须一起纳入切换。

## 10. 测试与验证策略

### 10.1 开发方式

本项目规则要求先完成开发实现，再统一编写测试脚本和执行测试。  
但本次属于行为改造，仍要保持最小可验证闭环。

实现阶段至少应补足下面一种验证方式：

- 新增与文案切换直接相关的前端单元测试
- 或在现有工具约束下，用类型级和构建级验证覆盖本次改动

### 10.2 最低验证要求

至少执行：

- `npm run lint`
- `npm run build`

如果实现中补了前端测试，则还需要执行对应测试命令。

### 10.3 验收标准

本次验收以以下结果为准：

1. 控制台默认进入中文
2. 顶部存在明确可见的语言切换按钮
3. 点击按钮后，所有主要可见 UI 文案切换为英文
4. 左上角品牌区显示 logo 和 `LlmNode`
5. 页面中不再出现 `Nexus Console`
6. 页面中不再保留大面积中英混杂的硬编码文案
7. 构建和类型检查通过

## 11. 文档回流判断

这次改造主要影响前端展示层，不改后端契约和流程。  
通常不需要同步：

- `docs/contracts/*.md`
- `docs/process/*.md`
- `docs/blueprint/current.md`

除非实现过程中顺带改变了：

- `README.md` 中的前端入口说明
- 控制台对外定位或最小启动方式

否则本次正式文档落点以本设计文档为主。

## 12. 风险与控制

### 12.1 风险

- 文案点位分散，容易遗漏
- 页面内有状态值、按钮、空态、placeholder 等细碎文本，人工替换时容易不一致
- 当前工作树已存在其他 `web-console` 相关未提交改动，不能误覆盖

### 12.2 控制措施

- 实现前先确认目标文件当前内容
- 使用集中词典，减少遗漏
- 修改时只触达本次范围内的前端文件
- 验证时对 `Nexus`、英文遗留词和中文遗留词做关键字回归搜索

## 13. 后续增强

以下内容不纳入本次范围，但后续可考虑：

- 记住用户上次语言选择
- 抽离更完整的 `useI18n` hook
- 为后端返回的部分状态值建立更系统的显示词典
- 增加 E2E 截图回归，覆盖中英文两种语言
