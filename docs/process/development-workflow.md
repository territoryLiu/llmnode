# 开发流程

## 0. 文档定位

这份文档描述 `llmnode` 的开发、迭代和文档回流流程。  
它不回答“系统现在是什么”，也不回答“未来想做什么”，而是回答：

- 一个需求进来后，先看哪里
- 什么时候可以直接开发
- 什么时候要先展开设计
- 开发完后要把哪些正式文档同步回去

如果只记一条规则，可以记这一句：

- `roadmap` 决定“现在该不该做”
- `spec` 决定“准备怎么设计”
- `plan` 决定“接下来怎么执行”
- `current / contracts / process` 决定“落地后系统现在是什么”

它不负责：

- 定义 AI 工作规则，那是 `AGENTS.md`
- 定义当前系统是什么，那是 `docs/blueprint/current.md`
- 定义未来优先级，那是 `docs/blueprint/roadmap.md`

## 1. 开发前先看什么

### 1.1 先判断这次走哪一档流程

当前项目不要求所有任务都走同一套最重流程。默认先按复杂度判断：

- 轻流程  
  适用于文档小修、单文件小改、现有契约下补字段、不改主链路边界的改动
- 标准流程  
  适用于一般功能迭代、中等范围 bugfix、某条链路增强
- 重流程  
  适用于跨层改动、改主链路、改契约、改运行形态、改存储边界、需要迁移顺序与回滚点的任务

一句话原则：

- 项目流程决定“该不该做、先看什么、改完回哪里”
- 通用 skill 决定“这次用多重的方法做”

### 1.2 标准阅读顺序

真正开始开发前，建议至少按下面顺序扫一遍：

1. `README.md`
2. `docs/blueprint/current.md`
3. `docs/blueprint/roadmap.md`
4. 相关 `docs/superpowers/specs/*.md`
5. 相关 `docs/contracts/*.md`
6. 相关 `docs/process/*.md`
7. 必要时看 `docs/knowledge/*.md`

这个顺序的意义是：

- 先确认项目入口和规则
- 再确认当前真相
- 再确认这件事是不是当前优先项
- 再确认有没有现成设计
- 最后才进入契约、流程和实现

### 1.3 什么时候看 `roadmap`

下列情况先看 `docs/blueprint/roadmap.md`：

- 你还不确定这件事是不是现在该做
- 有多个需求同时存在，需要判断优先级
- 你知道要扩功能，但还不知道当前阶段最该补哪层
- 你想确认某项能力是 `P0 / P1 / P2` 哪一档

`roadmap` 解决的是“该不该现在做”。

### 1.4 什么时候看或补 `spec`

下列情况优先看或补 `docs/superpowers/specs/*.md`：

- 这个需求已经有现成设计稿
- 你要做的是跨多层变更，不只是单点修补
- 这件事涉及迁移顺序、回滚点、兼容策略
- 你不只是要改结果，还要改边界
- 你需要把“为什么这样设计”沉淀成后续可复用文档

`spec` 解决的是“准备怎么设计”。

### 1.5 什么时候补 `plan`

下列情况建议在 `spec` 之后再补 `docs/superpowers/plans/*.md`：

- 同一设计要拆成多个实现阶段
- 需要明确实施顺序、检查点和完成条件
- 需要把风险拆散到多轮交付里
- 需要给后续实现或协作者一个稳定执行清单

`plan` 解决的是“接下来怎么执行”。

### 1.6 什么时候可以直接开发

下面这些情况通常可以不先写 `spec / plan`，直接开发：

- 小范围文案修订
- 现有契约下的局部补充
- 不改变主链路边界的字段补齐
- 单一文件内的轻量结构调整

但即使直接开发，也仍然要先确认：

- 它是不是当前优先项
- 它影响哪些正式产物
- 它落地后哪些文档要同步

## 2. 需求进入后的第一轮判断

每个需求进来时，先回答三件事。

### 2.1 它属于哪一类

- 当前状态变更：影响主链路、正式产物、事实层或流程执行
- 未来规划变更：影响优先级、排序、延后项或明确不做项
- 方案设计变更：影响架构边界、迁移顺序、兼容策略
- 契约变更：影响命令语义、输出结构、字段语义或状态表达
- 流程变更：影响运行、开发、部署、验收或回流顺序
- 知识沉淀变更：影响选型背景、术语解释或工程经验说明

### 2.2 它影响哪些层

至少回答下面几个问题：

- 影响哪类正式产物
- 影响哪条链路
- 影响哪份契约
- 会不会改变“现在是什么”
- 会不会改变“下一步做什么”
- 会不会改变“怎么验收”

如果需求涉及 gateway 三协议入口、route 分发、工具协议或请求透传，还要额外先判断三件事：

- 这次变更属于原生协议路径，还是跨协议 adapter 路径
- route 是否已经显式声明该协议原生支持或 adapter 允许
- 这次实现是否会引入 payload 语义改写；如果会，是否能被明确记录为 adapter / mutation，而不是静默兼容

### 2.3 这次主要真相源在哪

开发前先确认这次改动的真相源主要落在哪：

- 代码行为
- 配置文件
- 契约文档
- 流程文档
- 当前蓝图
- 设计稿 / 执行计划

如果真相源没找清楚，后面最容易出现“代码改了，但文档不知道该改哪”的问题。

## 3. 文档对象如何分工

不要默认所有任务都要同时产出全部文档对象。先判断真正需要的是哪一类：

- `roadmap`  
  解决“这件事现在该不该做、优先级如何”
- `spec`  
  解决“这次准备怎么设计、为什么这样设计、边界怎么定”
- `plan`  
  解决“接下来按什么顺序做、检查点是什么”
- `knowledge`
  解决“为什么这样做、有哪些经验和背景可参考”，但不承担当前状态、正式契约或正式流程真相
- `current / contracts / process`  
  解决“改完以后系统现在是什么”

默认原则：

- `spec` 与 `plan` 是设计与实施过程文档
- `knowledge` 是常驻参考层，不是正式真相层
- `current / contracts / process` 才是长期真相回流层
- 长期有效的信息最终仍应回流到 `blueprint / contracts / process`

## 4. 开发前 Checklist

开始动手前，建议逐项确认：

- 这件事是不是当前优先项
- 它有没有现成 `spec`
- 如果有 `spec`，这次实现覆盖 `spec` 的哪一部分
- 是否需要补 `plan`
- 会影响哪些正式产物
- 会影响哪些 `docs/contracts/*.md`
- 会影响哪些 `docs/process/*.md`
- 会不会改变 `docs/blueprint/current.md` 的描述
- 会不会让 `docs/blueprint/roadmap.md` 某项完成、降级或移动优先级
- 会不会改变阶段判断，需要补 `history.md`
- 需要什么验证方式

如果其中两项以上回答不出来，最好不要直接开始写代码，先把边界补清楚。

## 5. 标准开发顺序

推荐的默认顺序不是“先改代码再补文档”，而是下面这样：

1. 先确认当前状态与优先级
2. 再确认现有 `spec` 或补 `spec`
3. 必要时补 `plan`
4. 再看相关契约和流程
5. 再实现代码改动
6. 再回流文档
7. 最后统一验证

### 5.1 如果已有 `spec`

建议顺序：

1. 读 `roadmap`，确认它仍然是当前优先项
2. 读 `spec`，确认本次实现范围
3. 如实现步骤复杂，再读或补 `plan`
4. 读相关契约和流程，确认落地点
5. 改代码
6. 按 `spec / plan` 回流正式文档

### 5.2 如果没有 `spec`，但影响面较大

建议顺序：

1. 先补 `spec`
2. 明确迁移步骤和回滚点
3. 必要时补 `plan`
4. 再进入实现

### 5.3 如果需求涉及多协议网关

默认执行判断应按下面顺序：

1. 先确认 route 的 `native_protocols`
2. 再确认 route 的 `adapter_policies`
3. 再确认 `tool_policies / protocol_features`
4. 原生支持则优先走 `native pass-through`
5. 非原生支持时，只有显式 adapter 才允许转换
6. 如果既不原生支持、也无显式 adapter，应按保守失败处理，而不是补隐式兼容

### 5.3 如果是小改动

建议顺序：

1. 确认优先级
2. 直接读 `current + contracts + process`
3. 改代码
4. 同步必要文档

## 6. 开发完成后的文档回流

开发完成后，文档更新不是可选项，而是交付的一部分。

### 6.1 什么时候更新 `current.md`

下列情况通常应更新 `docs/blueprint/current.md`：

- 主链路发生实质变化
- 新的正式产物类型已经落地
- 真相源边界发生变化
- 原先在 `roadmap` 里的能力正式落地
- 当前阶段判断被改变

### 6.2 什么时候更新 `roadmap.md`

下列情况通常应更新 `docs/blueprint/roadmap.md`：

- 某项工作完成，应从未来规划移出
- 某项工作优先级改变
- 某项工作被明确延后
- 某项工作被明确取消

### 6.3 什么时候更新 `history.md`

下列情况通常应更新 `docs/blueprint/history.md`：

- 这次改动足以改变系统阶段判断
- 这次改动形成了新的版本级里程碑
- 一个长期设计已经落地，并改变了系统成熟度认知

### 6.4 什么时候更新 `spec / plan`

对 `docs/superpowers/specs/*.md` 与 `docs/superpowers/plans/*.md`，默认这样处理：

- 如果它们仍承担当前任务的设计或执行真相，就继续更新
- `docs/superpowers/*` 的默认定位是进行中工作区，不是长期沉淀库
- 如果其中有效内容已经沉淀为长期规则，就回流到 `blueprint / contracts / process`
- `plan` 如果只服务一次实现，应在任务完成且正式文档回流后删除
- `spec` 如果只服务一次设计展开，应在回流完成后删除；只有跨阶段仍在推进、短期仍需反复参考时才保留
- 不要强行把 `spec / plan` 升格为长期主入口

### 6.5 什么时候更新 `docs/knowledge/*`

对 `docs/knowledge/*`，默认这样处理：

- 它们用于保留选型背景、环境经验、联调 Q&A、转换说明和对比资料
- 它们可以长期保留，但不承担当前状态、正式契约、正式流程或未来优先级真相
- 如果某条知识已经变成必须遵守的正式约束，应回流到 `current / contracts / process`
- 如果某篇知识文档已经明显过时且没有参考价值，应删除或改写，而不是继续悬挂

### 6.6 什么时候更新 `README.md`

只有下面这些内容发生变化时，才需要同步更新 `README.md`：

- 项目总入口
- 最小启动方式
- 文档阅读顺序
- 关键边界说明

不要因为控制命令细节、局部流程步骤或实现内部结构变化，就把细节重新堆回 README。

## 7. 按变更类型看的文档联动

### 7.1 功能变更

通常要同步：

- `docs/blueprint/current.md`
- 相关 `docs/contracts/*`
- 相关 `docs/process/*`

### 7.2 架构或边界变更

通常要同步：

- `docs/blueprint/current.md`
- `docs/blueprint/history.md`
- 相关 `docs/superpowers/specs/*.md`
- 必要时更新 `docs/superpowers/plans/*.md`

### 7.3 契约变更

通常要同步：

- 对应 `docs/contracts/*.md`
- `docs/blueprint/current.md`
- 必要时同步 `docs/process/*.md`

### 7.4 流程变更

通常要同步：

- `docs/process/run.md`
- `docs/process/development-workflow.md`
- `docs/process/deployment.md`

### 7.5 控制面变更

通常要同步：

- `docs/contracts/control-plane.md`
- `docs/process/run.md`
- `docs/blueprint/current.md`
- 相关测试

## 8. 验证顺序

每次改完后，至少做三层验证。

### 8.1 结构验证

- 文档是否仍有统一入口
- 当前状态、历史和未来规划有没有混写
- `spec / current / roadmap` 有没有越界

### 8.2 内容验证

- 术语是否一致
- 结论是否仍能追溯到代码、配置、命令或样例
- 契约是否仍和真实行为一致

### 8.3 行为验证

- 主链路是否仍然能跑
- 命令入口是否仍可用
- 失败时是否能看懂原因
