# LLM 开发笔记

## 1. 先说结论

- 判断大模型能不能本地跑，不能只看 GPU 显存。
- 更关键的是两件事：
  - 模型权重能否由 RAM 和 VRAM 共同承载
  - VRAM 能否容纳热路径和 KV Cache
- 对 MoE 模型来说，“每个 token 只激活一部分参数”不等于“只需要存一部分权重”。

## 2. Dense 和 MoE 的差别

### 2.1 Dense 模型

- 每个 token 都会走完整参数路径。
- 一般特点：
  - 权重尽量常驻 GPU
  - CPU offload 会明显拉低速度
  - 更依赖显存容量和显存带宽
- 可以把它理解成“全员都要到场”。

### 2.2 MoE 模型

- 每个 token 只激活少数专家。
- 一般特点：
  - 总权重仍然很大
  - 计算路径更稀疏
  - 更适合分层存储和按需调度
- 可以把它理解成“人都在系统里，但每次只叫一部分出来干活”。

## 3. 真正占资源的是什么

### 3.1 权重存储

- 模型总权重必须有地方完整保存。
- 常见做法是：
  - GPU 放热路径或高频使用部分
  - CPU RAM 放完整权重或冷权重

### 3.2 激活路径

- 路由器会决定当前 token 走哪些 expert。
- 这些热路径通常更适合放在 GPU 上。

### 3.3 KV Cache

- 上下文一长，KV Cache 会非常吃显存。
- 很多“模型能加载但跑不动”的问题，最后卡住的其实是 KV Cache，不是权重本身。

## 4. 瓶颈怎么看

通常可以按下面这个顺序理解：

1. GPU 显存容量
2. PCIe 带宽
3. RAM 容量
4. 当前激活参数规模

这里要分开看两件事：

- 能不能跑，主要看权重和 KV Cache 能不能放下。
- 跑得快不快，才更多取决于激活参数规模和数据搬运带宽。

## 5. 单卡 24GB 的现实边界

下面以 24GB 显存的 4090 为例。

- 它不能让 35B 级 FP16/BF16 权重完整常驻 GPU。
- 但它可以通过 GPU + CPU 的混合部署运行一部分大模型，尤其是更适合 offload 的 MoE 模型。
- 现实前提通常是：
  - RAM 足够大
  - 上下文长度别太激进
  - 可以接受一定延迟

### 5.1 这类机器更适合做什么

- 4K 到 8K 上下文的本地推理
- CPU offload 场景
- 本地 Agent 或开发助手
- 低比特量化推理

### 5.2 这类机器通常不适合做什么

- 35B FP16 全量常驻 GPU
- 高并发、满血的 `vLLM` 服务化部署
- 超长上下文下同时追求高吞吐

## 6. 后端框架怎么选

### 6.1 `llama.cpp`

- 更适合混合部署和 CPU offload。
- 比较适合：
  - 单卡 24GB
  - MoE 模型
  - 本地开发、Agent、实验性部署

### 6.2 `vLLM`

- 更偏向 GPU 常驻、服务化和高吞吐。
- 在“显存不够、需要强 offload”的场景里，通常不如 `llama.cpp` 顺手。

### 6.3 `MLX`

- 如果是 Apple Silicon 的统一内存机器，体验通常会更好。
- 更适合统一内存前提下的本地推理。

## 7. 部署建议

### 7.1 单 4090

- 更推荐 MoE 加 CPU offload。
- 配置思路通常是：
  - RAM 尽量大
  - `ctx` 不要开得过猛
  - 优先保稳定，再谈极限速度

### 7.2 更理想的硬件

- 双 4090
- 更大显存单卡
- 大统一内存的 Apple Silicon 机器

## 8. 模型选择的工程经验

### 8.1 `Qwen3.6-35B-A3B`

- 更偏 coding、agent、tool use。
- 更适合本地开发和自动化工作流。
- 对低显存混合部署相对友好。

### 8.2 `Gemma 4 31B`

- 更偏聊天体验、视觉和 dense reasoning。
- 更适合日常对话和视觉任务。

> 注：这一节是工程经验，不是最新榜单或 benchmark 结论。涉及具体分数、发布日期、官方能力边界时，应单独核验。

## 9. 选型建议

- 目标是 coding、agent、工具调用：优先考虑 `Qwen3.6`
- 目标是更自然的对话体验：优先考虑 `Gemma 4`
- 目标是低显存本地部署：优先考虑 `Qwen3.6`
- 目标是多模态或视觉任务：优先考虑 `Gemma 4`

## 10. 一句话总结

- Dense 是“全员到场”。
- MoE 是“按需叫人”。
- 真正决定能不能跑的，不是激活参数本身，而是权重可访问性、显存容量、带宽和 KV Cache。









- 目标模型：models/deepseek/DeepSeek-V4-Flash
  - 目标机器：单卡 A800 80GB
  - 目标策略：尽量把大部分权重放 CPU 内存，GPU 主要承接活跃计算
  - 量化方向：Q4
  - 你的机器主存很充裕：503GiB RAM，当前可用约 474GiB

  这个方向不是胡思乱想，理论上是成立的。但要分清三件事：

  1. “能加载起来”
  2. “能稳定提供服务”
  3. “在当前 llmnode 这套控制面里能维护地上线”

  这三件事不是一回事。

  结论先说
  我不建议你在当前仓库里优先走 vLLM + CPU offload + DeepSeek-V4-Flash 这条线。
  我建议的顺序是：

  1. 先走 llama.cpp + GGUF(Q4) 路线做可启动验证
  2. 把目标先定成“短上下文可服务”，不要一开始追长上下文
  3. 如果启动和单条推理都稳定，再决定要不要把它纳入 config/backends 正式 profile

  原因很硬：

  - DeepSeek-V4-Flash 官方模型卡是 284B total / 13B activated / 1M context 的 MoE
  - 这类模型“激活参数少”不等于“整模型部署成本低”
  - 你想要的“主存放权重、GPU只跑活跃部分”更接近 llama.cpp 的传统 offload 思路
  - 当前仓库的 vLLM 容器参数入口里没有 cpu_offload_gb 之类字段，控制面也没接这类参数
  - 所以 vLLM 路线即使理论可做，在这个项目里也还不是“拿来就能进正式路径”的状态

  为什么我推荐 llama.cpp + GGUF Q4
  对你这台机器，它是最符合硬件现实的：

  - 你有 80GB VRAM，但不够承载这种级别模型的“舒服部署”
  - 你有 503GiB RAM，这才是这条路线真正的缓冲池
  - llama.cpp 天生更适合：
      - GGUF 量化
      - 权重主要驻留主存
      - 部分层/算子下沉到 GPU
      - 用 n_gpu_layers 控制下卡规模

  换句话说，llama.cpp 更像你说的“模型加载到内存，激活层推理在卡上”。

  三种可选方案
  我按你当前仓库和硬件，给你三个方案：

  1. 推荐：llama.cpp + GGUF Q4_K_M / Q4_K_XL
      - 目标：先跑通、先稳定
      - 权重大头在 RAM，GPU 做加速
      - 最符合你现在的单机资源结构
      - 风险：速度不会特别漂亮，但最容易落地
  2. 次选：vLLM + 官方权重 + CPU offload
      - 目标：尽量保留官方推理路径
      - 问题：你这套仓库控制面现在没接 cpu offload 相关参数
      - 风险：需要扩配置、扩容器参数、重新验证控制面契约
      - 这不是“直接部署”，而是“先改平台能力”
  3. 不推荐现在做：追求长上下文正式部署
      - DeepSeek-V4-Flash 官方主打 1M context
      - 单卡本地即使 Q4，也不该先拿长上下文当目标
      - 会把问题从“能不能跑”变成“KV cache 和吞吐全炸”

  我建议的部署目标
  如果你现在就是想让它在这台机器上先可用，我建议把目标定成：

  - 后端：llama.cpp
  - 格式：GGUF
  - 量化：优先 Q4_K_M，如果有更合适社区量化再看
  - 上下文：先从 16K 或 32K 起
  - 并发：1
  - n_gpu_layers: -1 先试；如果不稳，再降成部分下卡
  - 验收标准：
      - 容器稳定启动
      - /v1/models 可达
      - 单条短请求稳定返回
      - 再逐步加上下文

  当前阻塞点
  你现在不是先改 profile 的时候，先要确认两件事：

  1. DeepSeek-V4-Flash 有没有成熟可用的 GGUF Q4 版本
  2. 这个 GGUF 版本是否已经有人验证能在 llama.cpp 上本地推理

  如果这两件事不成立，后面全是空转。
